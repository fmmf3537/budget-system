## 需求一致性检查（可能不通顺/歧义点）

### 1）“资金计划按照月度生成” vs “月度子计划”
- 目前描述像是两层结构：
  - **月度计划（主计划）**：由 admin 生成，代表某组织某月份的“汇总现金计划/基准计划”。
  - **月度子计划**：由非 VIEWER 的用户创建并提交，代表某个部门/项目/维度范围内的“本月现金计划”。
- 需要明确：**子计划是否必须归属到某个月度主计划**？（建议：必须。否则“合并到月度计划”没有归宿）同意建议
- 需要明确：主计划生成后是否允许 admin 自己也创建子计划？（建议：允许，只要角色非 VIEWER 且 active）同意建议

### 2）“所有月度子计划提交后自动给生成该月度计划的 admin 用户进行审批”
- 这句话可能存在两个理解：
  - **理解 A：逐个子计划审批**：每个子计划提交后，都由该 admin 审批（不需要等全部提交）。
  - **理解 B：汇总后审批**：要等“所有子计划都提交”才触发审批（需要先定义“所有”的范围和截止点，很难实现且易争议）。
- 建议采用 **理解 A（逐个子计划提交即触发待办）**，并补充“当月是否有截止日/锁定点”的业务规则（见下文）。同意建议

### 3）“月度子计划审批通过后，合并到月度计划中”
- 需明确“合并”语义，否则容易引起口径不一致：
  - 合并到底是 **按日汇总** 还是 **按类别汇总** 还是 **明细逐行追加**？
  - 同一子计划如果被撤回/驳回/重新提交，月度主计划如何回滚或重新计算？
- 建议：主计划不做“累加式写死”，而是 **可重复计算的汇总视图/汇总表**：
  - 月度主计划的“合并结果”始终由 **当月所有已 APPROVED 的子计划明细汇总** 计算得到（避免重复合并与回滚困难）。同意建议

### 4）“余额只有 admin 可见”
- 需要明确“余额的部分”具体包含哪些字段/指标：
  - 期初余额、期末余额、预测结余、最低余额、现金安全水位、历史预测对比等？是的
- 需要明确“不可见”是：
  - 直接不返回 API 字段（推荐，防止抓包看到），还是前端隐藏但 API 仍返回（不安全）？选择不返回API字段
- 还需明确：非 admin 是否仍能看到“净流入/净流出/总流入/总流出”等汇总？（一般可见，否则子计划对齐困难）可见

### 5）用户范围与状态（active）
- “非 VIEWER 的 active 状态用户”需要进一步明确：
  - active 指 `User.status=ACTIVE`？是的
  - 非 VIEWER 包括 APPROVER 吗？包括 ADMIN/BUDGET_MANAGER 吗？是的
  - 如果用户是 APPROVER，但不是预算编制人员，是否应该允许建子计划？（建议：用权限码控制，而不是角色名）同意建议

## 建议补全的业务对象与字段（防止开发时各自理解）同意建议

### A. 月度主计划（Monthly Cash Plan / 主计划）
- **唯一性**：同一组织（organizationId）+ 月份（YYYY-MM）应当最多 1 个主计划（避免重复生成）。
- **创建方式**：仅 admin 点击“生成月度计划”按钮创建（必要时可覆盖/重建，见异常流程）。
- **基础字段**：
  - `organizationId`
  - `month`（建议存为月初日期或 `YYYY-MM` 字符串）
  - `createdByAdminUserId`
  - `status`：建议至少 `DRAFT/OPEN`、`LOCKED/CLOSED`（主计划是否允许继续接收子计划）
  - `note/remark`（可选）
- **权限**：
  - 生成按钮：仅 admin（或 `CASH_PLAN_MONTHLY_GENERATE` 权限）
  - 查看：非 VIEWER 可看主计划基本信息；余额字段仅 admin 可看（见下文“字段可见性”）

### B. 月度子计划（Monthly Cash Sub-Plan / 子计划）
- **归属关系**：必须关联到某个主计划：`parentMonthlyPlanId`
- **建议唯一性（可选但很关键）**：
  - 如果业务希望“每个部门每月只有一份子计划”，则需 `unique(parentId, departmentCode)`；
  - 若希望“按维度拆分”，需要定义拆分口径，比如 `unique(parentId, departmentCode, dimension1, dimension2)`；
  - 若希望“一个人可多次提交版本”，则引入 `version` 或保留历史（否则覆盖会丢审计）。
- **状态流转**（建议）：
  - `DRAFT`（可编辑）
  - `SUBMITTED`（提交后不可编辑）
  - `APPROVED`（审批通过）
  - `REJECTED`（驳回后回到 DRAFT 或保持 REJECTED 但允许编辑，需二选一并一致）
  - `WITHDRAWN/CANCELLED`（提交人撤回，待办取消）
- **提交与审批**：
  - 提交动作：子计划 `DRAFT -> SUBMITTED`，并为主计划创建人 admin 生成审批待办（单个子计划维度）
  - 驳回动作：`SUBMITTED -> DRAFT`（推荐回到草稿，方便继续改；同时保留驳回记录）
  - 同意动作：`SUBMITTED -> APPROVED`

## 合并逻辑（“审批通过后合并到主计划”）推荐方案 同意建议

### 推荐：主计划汇总=实时聚合，不做不可逆写入
- 主计划的“合并结果”由查询聚合得到：
  - 取 `parentMonthlyPlanId = X` 且 `status = APPROVED` 的所有子计划明细
  - 按（日期、类别、方向）或（类别、方向）进行汇总，形成主计划展示
- 好处：
  - 子计划被撤回/重审/作废时，主计划结果自动一致
  - 避免重复“合并写入”导致的 double count

### 若必须“写入式合并”（不推荐）则必须补齐
- 合并幂等（同一子计划只能合并一次）
- 反合并/回滚（子计划被撤回/驳回后如何扣回）
- 并发控制（同时审批多个子计划）

若业务真的需要“全部提交后才开始审批/才生成待办”，必须先定义：
- “全部”的范围是什么？按部门列表？按维度组合？按人员名单？（这些都可能变化）
- 截止时间是谁设置？到点未提交怎么办？

## 资金计划（月度主计划/子计划）详细功能说明（v2：结合确认意见后的可验收规格）

> 本节是在你对上述建议的“同意/补充”基础上，将需求重新梳理为可实现、可测试、可验收的版本。

### 0. 名词与目标
- **月度主计划**：某组织某自然月（本地日历月）的现金计划“容器/汇总视图”，由 **ADMIN** 生成。
- **月度子计划**：在某主计划下，由业务人员提交的现金收支计划明细，提交后由主计划创建的 ADMIN 审批。
- **合并**：当子计划 **审批通过** 后，其明细进入主计划汇总口径（推荐通过聚合计算实现，确保可回滚与幂等）。
- **余额区块**：期初/期末/安全水位/预测余额曲线等，仅 ADMIN 可见，且 **非 ADMIN 请求接口时后端不返回这些字段**。

### 1. 用户与权限（验收口径）
- **可创建/提交子计划的用户范围**：
  - `User.status = ACTIVE`
  - 角色为 **非 VIEWER**（包括 ADMIN / BUDGET_MANAGER / APPROVER），但建议最终以权限码落地，避免角色变化引起偏差。
- **主计划生成**：仅 ADMIN（或具备 `CASH_PLAN_MONTHLY_GENERATE` 权限）。
- **余额查看**：仅 ADMIN（或具备 `CASH_PLAN_BALANCE_VIEW` 权限）。
- **审批处理人**：固定为该主计划的创建 ADMIN。
  - 若该 ADMIN 非 ACTIVE / 不存在：由 **系统管理员接管**（需要定义接管触发方式，见第 7 节）。

### 2. 时间口径与日期校验（本地日历月）
- **月度范围**按本地日历月定义，例如 2026-04 的范围为：
  - 起始：2026-04-01 00:00:00（本地）
  - 结束：2026-04-30 23:59:59（本地）
- **子计划明细行日期**必须落在其归属月份内，否则拒绝保存/提交（返回校验错误）。
- 所有接口/页面展示日期以本地日历月理解为准（避免 UTC 偏移导致跨月）。

### 3. 金额规则（避免歧义）
- 子计划的每一条收入/支出明细行金额：
  - **必须为正数**（确认：不可为负数或 0）
  - 小数精度建议固定为 2 位（或与系统货币精度一致）
- 若需要表达“冲销/调整”，必须通过：
  - 单独的“冲销类”类别（正数）+ 备注说明，或
  - 走预算调整/子计划撤回重提流程（不允许负数抵消）。

### 4. 数据结构与唯一性（必须写死，否则实现会跑偏）

#### 4.1 月度主计划（MonthlyPlan）
- 唯一性：`unique(organizationId, month)`，同月只能存在一个主计划。
- 字段（建议最小集）：
  - `organizationId`
  - `month`（YYYY-MM 或 monthStartDate）
  - `createdByAdminUserId`
  - `status`: `OPEN | LOCKED`（OPEN 允许创建/提交子计划；LOCKED 禁止新建/提交）
  - `createdAt / updatedAt`

#### 4.2 月度子计划（SubPlan）
- 必须归属：`parentMonthlyPlanId`
- 字段（建议最小集）：
  - `organizationId`
  - `parentMonthlyPlanId`
  - `ownerUserId`（创建/负责用户）
  - `scopeDepartmentCode`（可选但强烈建议：若不引入范围字段，“子计划”很难定义“谁该提交什么”）
  - `scopeDimension1 / scopeDimension2`（可选，用于按维度拆分口径）
  - `status`: `DRAFT | SUBMITTED | APPROVED | REJECTED | WITHDRAWN`
  - `submittedAt / approvedAt`
- **唯一性口径（建议采用，便于验收）**：
  - 若以部门为提交单位：`unique(parentMonthlyPlanId, scopeDepartmentCode)`
  - 若以部门+维度为提交单位：`unique(parentMonthlyPlanId, scopeDepartmentCode, scopeDimension1, scopeDimension2)`
- 若不做唯一性，则必须明确“允许同一范围多份子计划”的排序与汇总规则（强烈不建议）。

### 5. 状态流转与操作（核心验收点）

#### 5.1 主计划
- `OPEN`：
  - 允许非 VIEWER active 用户创建/编辑/提交子计划（编辑仅限子计划 DRAFT）
  - 允许 ADMIN 查看余额区块
- `LOCKED`：
  - 禁止创建新子计划
  - 禁止子计划从 DRAFT 提交（已有 SUBMITTED/APPROVED 仍可查看与审批按规则执行或一并冻结，需二选一）

#### 5.2 子计划
- **创建**：DRAFT
- **提交**：DRAFT → SUBMITTED
  - 触发：自动给主计划创建 ADMIN 生成审批待办（逐个子计划触发，不等“全部提交”）
- **撤回提交**：SUBMITTED → DRAFT 或 WITHDRAWN（二选一）
  - 确认：撤回时取消待办（审批记录 PENDING → CANCELLED）
  - 限制：若该子计划审批链上已出现 APPROVED，则禁止撤回
- **驳回**：SUBMITTED → DRAFT（确认：驳回后允许继续编辑并再次提交）
- **同意**：SUBMITTED → APPROVED
  - 同意后进入主计划汇总口径（见第 6 节）

### 6. “合并到主计划”的计算口径（确认采用聚合式）
- 主计划的汇总数据 = 该主计划下所有 **APPROVED 子计划**明细的聚合结果。
- 聚合维度（建议明确，否则报表对不上）：
  - 最小：按（收支方向 IN/OUT + 类别 code）汇总；
  - 若需要现金流预测：再按（日期 + 收支方向 + 类别）汇总。
- 非 ADMIN 查看主计划时：
  - 可见：总流入/总流出/净流量（来自聚合）
  - 不可见：期初/期末/预测余额曲线/安全水位等（后端不返回）

### 7. 审批与接管规则（避免 admin 不可用导致卡死）
- 默认审批人 = 主计划创建 ADMIN。
- 若该用户 `status != ACTIVE` 或不存在：
  - 系统选择“系统管理员接管”：
    - 方案 1（推荐）：由 ADMIN 角色用户中选第一个（或指定的 `superAdmin`）作为接管人
    - 方案 2：允许在“主计划详情”里手动变更审批接收人（审计记录）
- 接管后，新提交子计划的待办均发给接管人；已存在待办可批量转交或按规则重新指派。

### 8. 删除规则（你已确认主计划可删，且需级联）

#### 8.1 主计划删除（仅 ADMIN）
- 允许删除主计划（建议仅 `OPEN` 状态可删；若 `LOCKED` 需二次确认）。
- 删除主计划时必须级联删除：
  - 所有子计划及其明细
  - 所有关联审批记录（ApprovalRecord）
- 删除前需校验：若存在 APPROVED 子计划，是否仍允许删（建议允许但强提示，避免误删造成数据断层）。

#### 8.2 子计划删除
- 建议仅 DRAFT 可删；SUBMITTED/APPROVED 不允许直接删（走撤回/驳回/作废流程）。

### 9. 接口与页面清单（便于任务拆分/验收）

#### 9.1 页面
- 主计划列表（ADMIN）：选择月份 → 生成；进入主计划详情
- 主计划详情：
  - 非 ADMIN：汇总（流入/流出/净流量）+ 子计划列表
  - ADMIN：在上述基础上额外展示余额区块（期初/期末/安全水位/预测）
- 子计划创建/编辑页（非 VIEWER active）：仅 DRAFT 可编辑；提交后只读；支持撤回
- 审批待办页（ADMIN/接管人）：逐个子计划同意/驳回/转交

#### 9.2 接口（示例，实际路径可按现有项目规范落地）
- 主计划：
  - `POST /api/cash-plan/monthly/generate?month=YYYY-MM`
  - `GET /api/cash-plan/monthly?month=YYYY-MM`
  - `DELETE /api/cash-plan/monthly/{id}`（级联删除）
- 子计划：
  - `POST /api/cash-plan/monthly/{monthlyId}/subplans`
  - `PUT /api/cash-plan/subplans/{id}`（仅 DRAFT）
  - `POST /api/cash-plan/subplans/{id}/submit`
  - `POST /api/cash-plan/subplans/{id}/withdraw`
  - `DELETE /api/cash-plan/subplans/{id}`（仅 DRAFT）
- 审批：
  - `POST /api/approval/{processId}/approve`（现有）
  - `POST /api/approval/{processId}/reject`（现有）
  - 需确保 entityType/entityId 绑定到“子计划”而非主计划（避免审批对象混淆）

## 功能补充：主计划按「顶级部门」限定范围（v3 增量）

### 1. 业务规则（必须写死）
- 每个月度主计划可选 **一个顶级部门**（部门树的根节点，`parentId = null`）。
- 若主计划 **不选择顶级部门**：默认为对 **整个组织** 的计划（全组织主计划）。
- 主计划 **不可选非顶级部门**（`parentId != null` 的节点禁止选择）。
- 子计划选择部门时需满足主计划范围：
  - 若主计划为全组织：子计划可选择任意部门（建议仍要求部门编码必须存在于部门主数据）。
  - 若主计划选择了顶级部门 A：子计划仅可选择 **A 子树内** 的部门。
    - 建议允许选择 A 本身（顶级部门也属于其范围），否则顶级部门负责人无法提交汇总子计划。

### 2. 数据结构建议（便于实现与验收）

#### 2.1 主计划增加字段
- `rootDepartmentCode: string | null`
  - `null` 表示全组织
  - 非空必须对应一个顶级部门的 `code`

#### 2.2 子计划字段
- 子计划需要有明确的部门范围字段（若已有则沿用）：
  - `scopeDepartmentCode: string`（或命名为 `departmentCode`）

### 3. 接口校验（后端必须强制）

#### 3.1 主计划创建/更新（ADMIN）
- 传入 `rootDepartmentCode` 时：
  - 必须属于当前 `organizationId`
  - 必须存在且 `parentId = null`
  - 否则返回 `VALIDATION_ERROR`

#### 3.2 子计划创建/更新（非 VIEWER active）
- 读取 `parentMonthlyPlan.rootDepartmentCode`：
  - 若为 `null`：允许任意存在的部门编码
  - 若为 `A`：仅允许 `scopeDepartmentCode` 属于 A 子树
    - 子树判定建议采用“向上追溯 parentId 链直到根节点”的方式：
      - 若最终根节点 `code === A`，则合法
      - 否则非法

### 4. UI/交互（可验收）

#### 4.1 主计划生成/编辑（ADMIN）
- 增加「顶级部门（可选）」下拉：
  - 下拉仅展示 `parentId = null` 的部门
  - 提供选项「全组织（不选择）」= `null`
- 主计划详情展示范围：
  - `rootDepartmentCode = null`：范围=全组织
  - 非空：范围=顶级部门名称（或 code + 名称）

#### 4.2 子计划创建/编辑页
- 部门下拉选项随主计划范围变化：
  - 全组织：展示全部部门
  - 顶级部门 A：展示 A 子树部门
- 若用户通过篡改请求选择越界部门：
  - 接口拒绝并提示“所选部门不在主计划范围内”

### 5. 唯一性与“同月多主计划”需要提前定版（避免返工）
- 若允许同月按不同顶级部门分别建主计划，则主计划唯一性应为：
  - `unique(organizationId, month, rootDepartmentCode)`（其中 `null` 也要作为一种取值参与唯一性）
- 若不允许，则保持 `unique(organizationId, month)`，仅作为主计划内部范围字段使用。
- **建议：优先支持同月多主计划**（全组织 + 分顶级部门），但必须明确汇总/报表展示规则；若暂不做，则明确“同月仅 1 份主计划”。

### 6. 新增验收用例
9. ADMIN 生成主计划时选择非顶级部门：失败并提示“仅可选择顶级部门”。
10. 主计划选择顶级部门 A 后：子计划部门下拉仅含 A 子树；选其它部门提交会被拒绝。
11. 主计划不选顶级部门（全组织）：子计划可选任意部门并正常提交审批。

## 最终验收规格（v4：单一口径、可实现/可测试/可验收）

> 本节将 v1~v3 的确认点合并为**唯一实施口径**。后续开发、测试、验收均以本节为准；若与上文冲突，以本节为准。

### 1. 目标与范围
- 将“资金计划”升级为**按月度主计划 + 月度子计划**的协同编制与审批机制。
- 支持：
  - ADMIN 生成月度主计划（可选顶级部门范围）
  - 非 VIEWER 且 ACTIVE 的用户创建/提交子计划
  - 子计划提交后自动给主计划创建 ADMIN 审批
  - 子计划审批通过后计入主计划汇总（聚合式合并）
  - 余额/预测等敏感信息仅 ADMIN 可见（非 admin 接口不返回）
  - 主计划可删除且级联删除子计划/明细/审批记录

### 2. 名词定义
- **主计划**：MonthlyPlan。某组织某月的现金计划汇总容器；可选择一个“顶级部门范围”或全组织范围。
- **子计划**：SubPlan。归属某主计划，由业务用户提交的本月现金收支明细。
- **合并**：主计划汇总 = 其下所有 APPROVED 子计划明细的聚合结果（不做不可逆写入累加）。

### 3. 组织/部门口径（新增顶级部门范围）
- 主计划字段：`rootDepartmentCode: string | null`
  - `null` 表示全组织
  - 非空必须对应 **顶级部门**（BudgetDepartment 记录满足 `parentId = null`）
- 主计划禁止选择非顶级部门作为 root。
- 子计划字段：`scopeDepartmentCode: string`（若已有同义字段沿用）
  - 若主计划 `rootDepartmentCode = null`：子计划可选择任意存在的部门编码
  - 若主计划 `rootDepartmentCode = A`：子计划只能选择 A 子树内部门（建议允许选择 A 本身）
- 后端强制校验部门范围，前端仅做下拉过滤不作为安全边界。

### 4. 时间口径（本地日历月）
- 月度范围按本地日历月定义（例如 2026-04：4/1 00:00:00 ~ 4/30 23:59:59，本地）。
- 子计划明细行日期必须落在主计划月份范围内；否则保存/提交失败。

### 5. 金额规则（强校验）
- 子计划收入/支出明细金额：
  - 必须为 **正数**（> 0），不允许 0 或负数
  - 精度按系统货币/统一保留 2 位小数（实现时统一处理）

### 6. 用户范围与权限（验收口径）
- 可创建/编辑/提交子计划的用户需同时满足：
  - `User.status = ACTIVE`
  - 角色非 VIEWER（包括 ADMIN / BUDGET_MANAGER / APPROVER）
  - 最终建议用权限码实现（如 `CASH_PLAN_SUBPLAN_CREATE/EDIT/SUBMIT`），但验收时以“非 VIEWER 且 ACTIVE”作为最低要求。
- 主计划生成/删除/余额查看：仅 ADMIN（或具备对应权限）。

### 7. 状态机与操作

#### 7.1 主计划状态
- `OPEN`：允许创建/提交子计划
- `LOCKED`：禁止新建子计划与 DRAFT→SUBMITTED 提交（是否允许审批进行中的 SUBMITTED 继续审批：建议允许；若禁止需额外定义冻结规则）

#### 7.2 子计划状态
- `DRAFT`：可编辑（含增删明细）
- `SUBMITTED`：只读，等待审批
- `APPROVED`：已通过，计入主计划汇总
- `REJECTED`：驳回后回到 DRAFT（本项目验收口径：驳回动作使其回草稿可继续修改再提交）
- `WITHDRAWN/CANCELLED`：撤回提交（可选实现为回到 DRAFT 或单独状态；验收至少要求“撤回后可再次编辑并可重新提交”）

#### 7.3 操作规则
- **提交子计划**：DRAFT → SUBMITTED
  - 提交后立即为审批人生成待办（逐个子计划触发，不等待“全部提交”）
- **撤回子计划**：仅 SUBMITTED 可撤回
  - 撤回时取消待办：审批记录 PENDING → CANCELLED
  - 若审批链上已出现 APPROVED（有人同意过），则禁止撤回
- **审批同意**：SUBMITTED → APPROVED
- **审批驳回**：SUBMITTED → DRAFT（同时保留审批记录为 REJECTED）

### 8. 审批规则（固定给主计划创建 ADMIN）
- 子计划提交后，审批对象为“子计划”（entityType/entityId 指向子计划，不指向主计划）。
- 默认审批人 = 创建该主计划的 ADMIN 用户。
- 若该用户不再 ACTIVE / 不存在：由系统管理员接管
  - 接管口径：选择组织内 ADMIN 用户作为审批接收人（需明确选择策略：指定 superAdmin / 第一个 ADMIN / 配置项）

### 9. 主计划汇总与展示（合并口径）
- 主计划汇总 = 所有 APPROVED 子计划明细的聚合结果。
- 聚合口径（至少一套要固定）：
  - 按（方向 IN/OUT + 类别）汇总得到本月总流入/总流出/净流量
  - 如需日级预测：按（日期 + 方向 + 类别）进一步汇总
- 非 ADMIN 可见：
  - 子计划明细、流入/流出/净流量汇总
- 仅 ADMIN 可见（且非 admin 接口不返回字段）：
  - 期初余额、期末余额、预测余额曲线、安全水位、余额预警等

### 10. 删除规则（强约束）

#### 10.1 主计划删除（仅 ADMIN）
- 允许删除主计划（建议仅 OPEN 可删；LOCKED 需二次确认）
- 删除时级联删除：
  - 所有子计划及其明细
  - 所有关联审批记录（ApprovalRecord）

#### 10.2 子计划删除
- 建议仅 DRAFT 可删除；SUBMITTED/APPROVED 不允许直接删除（走撤回/驳回/作废流程）

### 11. 接口与页面（用于拆分任务）
- 主计划（ADMIN）：
  - 生成：选择月份 +（可选）顶级部门
  - 删除：可见危险操作按钮/二次确认
  - 详情：展示范围、子计划列表；ADMIN 额外显示余额区块
- 子计划（非 VIEWER active）：
  - 在主计划下创建：部门下拉按范围过滤
  - DRAFT 编辑/提交；SUBMITTED 撤回；被驳回后再编辑提交
- 审批（ADMIN/接管人）：
  - 待办按子计划维度展示；支持同意/驳回/转交（若系统支持）

### 12. 最小验收用例（必须全部通过）
1. ADMIN 生成 2026-04 主计划：同月重复生成同范围应提示“已存在”（不生成重复）。
2. 主计划 rootDepartmentCode 只能选顶级部门；选非顶级会被拒绝。
3. rootDepartmentCode = A 时：子计划部门只能选 A 子树（含 A 本身）；越界提交被拒绝。
4. rootDepartmentCode = null 时：子计划可选任意部门并成功提交。
5. 子计划明细日期超出当月：保存/提交失败。
6. 子计划金额为 0 或负数：保存/提交失败。
7. 子计划提交后：立即给主计划创建 ADMIN 生成待办（不等待“全部提交”）。
8. ADMIN 同意后：子计划变为 APPROVED，主计划汇总立即反映该子计划。
9. 非 ADMIN 查看主计划：接口不返回余额相关字段，但可见流入/流出/净流量汇总。
10. SUBMITTED 子计划可撤回且取消待办；若已出现 APPROVED 记录则禁止撤回。
11. 主计划删除：子计划、明细、审批记录被级联删除。
